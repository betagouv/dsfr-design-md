DSFR · DESIGN.md — aperçu visuel

Cet aperçu rend le DESIGN.md du Système de Design de l'État en utilisant uniquement les tokens qu'il déclare. Si le fichier omet une valeur, le composant correspondant casse — c'est précisément l'objectif : utiliser cette page comme test de cohérence du DESIGN.md.

⚠️ Cadre d'usage. L'identité visuelle reproduite ci-dessous (Bleu France, fonte Marianne, bloc-marque République Française) est légalement réservée aux services officiels de l'État français. Pour toute mise en production sur un site .gouv.fr, utilisez le paquet npm officiel @gouvfr/dsfr. La fonte Marianne est chargée depuis le CDN officiel pour la fidélité typographique mais n'est pas redistribuée par ce dépôt.

Cette page est l'image miroir, en thème sombre, de preview.html. Les valeurs hexadécimales affichées correspondent au scheme dark du DSFR.

Couleurs

Tokens d'option du DSFR, regroupés par famille. Le code de chaque pastille indique le nom du token tel qu'il apparaît dans le YAML du DESIGN.md.

Pourquoi certaines teintes n'ont-elles pas de variantes -hover et -active ?

La distribution CSS officielle du DSFR (@gouvfr/dsfr/dist/dsfr.css) ne déclare des états interactifs que pour les nuances destinées à servir de surface d'action : la couleur primaire saturée (sun-113-625), les fonds d'action discrets (925-125, 950-100, 975-75), et les deux teintes interactives de chaque famille système (-425-625 et -950-100).

Les nuances suivantes ne portent pas d'états par construction, et c'est volontaire :

  • main et 850-200 : réservées aux bordures, filets et illustrations statiques.
  • Neutres (gris) : utilisés pour le texte et les fonds, jamais comme surfaces d'action autonomes.
  • -975-75 des familles système : fonds d'alertes non actionnables (un message d'erreur n'est pas un bouton).
  • Accents illustratifs : usage purement éditorial, jamais interactif.

Le DESIGN.md reflète exactement ce contrat : aucune teinte n'a d'-hover/-active que le DSFR canonique ne définirait pas, et inversement.

Tokens de décision

Les composants doivent référencer ces tokens, pas les options brutes. Chaque token de décision pointe vers un token d'option différent selon le thème (clair / sombre).
